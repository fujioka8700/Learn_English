# プロジェクト管理

## 概要

英語学習アプリのプロジェクト管理方法を記載します。

## プロジェクト情報

### プロジェクト名
英語学習アプリ (Learn English)

### プロジェクトの目的
中学1年〜3年レベルの英単語を効率的に学習できるWebアプリケーションを開発する。
誰でも登録なしで利用できることを前提とした設計。

### 対象ユーザー
- 中学生（中1、中2、中3）
- 英単語を基礎から学び直したい人
- 英単語の暗記を効率的に行いたい人
- 登録不要で気軽に使いたい人（ゲストユーザー）

## 開発フロー

### 1. ブランチ戦略

- **main**: 本番環境用ブランチ（安定版）
- **develop**: 開発用ブランチ（統合ブランチ）
- **feature/**: 機能開発用ブランチ
  - 例: `feature/quiz-function`
  - 例: `feature/user-authentication`
- **fix/**: バグ修正用ブランチ
  - 例: `fix/timer-bug`

### 2. コミットメッセージ規約

```
<type>: <subject>

<body>

<footer>
```

#### Type
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの動作に影響しない変更（フォーマットなど）
- `refactor`: リファクタリング
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

#### 例
```
feat: 4択クイズ機能の実装

- 4択問題の出題機能を追加
- カウントダウンタイマーの実装
- 結果画面の追加

Closes #123
```

### 3. プルリクエスト（PR）の流れ

1. 機能ブランチを作成
2. 開発・テスト
3. プルリクエストを作成
4. コードレビュー
5. マージ

## タスク管理

### タスクの優先度

1. **高**: 必須機能、ブロッカーとなる問題
2. **中**: 重要だが緊急ではない機能
3. **低**: 改善や最適化

### タスクの分類

- **機能開発**: 新機能の実装
- **バグ修正**: 既存機能の問題修正
- **リファクタリング**: コードの改善
- **ドキュメント**: ドキュメントの作成・更新
- **テスト**: テストの作成・実行

## 会議・レビュー

### 定例会議
- **頻度**: 週1回（必要に応じて調整）
- **内容**: 進捗確認、課題の共有、次のスプリントの計画

### コードレビュー
- すべてのプルリクエストはレビュー必須
- レビュー完了後にマージ

## ドキュメント管理

### 必須ドキュメント

1. **requirements.md**: 要件定義書
2. **tech-stack.md**: 技術スタック
3. **progress.md**: 進捗管理
4. **project-management.md**: プロジェクト管理（本ドキュメント）

### ドキュメント更新ルール

- 重要な変更があった場合は即座に更新
- 定期的な見直し（週1回推奨）
- 変更履歴を記録

## リスク管理

### 想定されるリスク

1. **技術的リスク**
   - 新しい技術の学習時間
   - パフォーマンスの問題
   - **対策**: 事前調査、プロトタイプ作成

2. **スケジュールリスク**
   - 開発時間の見積もり誤差
   - **対策**: バッファ時間の確保、優先度の明確化

3. **データリスク**
   - 英単語データの取得・インポートの問題
   - **対策**: データバックアップ、手動インポート手順の整備

## 品質管理

### コード品質

- TypeScript の型チェック
- ESLint によるコードチェック
- コードレビュー

### テスト

- 単体テスト（将来実装）
- 統合テスト（将来実装）
- 手動テスト

### パフォーマンス

- ページ読み込み時間: 2秒以内
- データベースクエリの最適化

## コミュニケーション

### 連絡手段

- GitHub Issues: タスク管理、バグ報告
- GitHub Discussions: 技術的な議論
- ドキュメント: 仕様や設計の記録

## リリース管理

### バージョニング

セマンティックバージョニング（Semantic Versioning）を使用

- **MAJOR**: 互換性のない変更
- **MINOR**: 後方互換性のある新機能
- **PATCH**: 後方互換性のあるバグ修正

例: `v1.0.0`, `v1.1.0`, `v1.1.1`

### リリースノート

各リリースで以下の情報を記載:
- 新機能
- バグ修正
- 破壊的変更
- 既知の問題
